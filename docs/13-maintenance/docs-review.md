# 🔎 Ревью документации (процесс)

> **Статус:** stable · **Версия:** 0.3 · **Дата:** 2026-07-23  
> **Назначение:** единый порядок ревизии `docs/` (и связанных meta-правил).  
> **Триггер для агента:** фразы вроде «ревью документации», «docs review», «ревизия docs».  
> **Связано:** [docs-guidelines.md](./docs-guidelines.md) · [DOCS-ROADMAP.md](../00-meta/DOCS-ROADMAP.md) · [AGENT-DOCS-INDEX.md](../00-meta/AGENT-DOCS-INDEX.md)

Каждая ревизия должна **повышать** качество документации, ясность процессов и согласованность docs↔код. Сами правила ревью — тоже объект улучшения.

---

## 🎯 Принципы

1. **Docs-first, не docs-only** — документация описывает намерение системы; код проверяется на соответствие, но ревью docs не подменяет code review.
2. **Слои по очереди** — сначала целостность и смысл docs, затем сверка с кодом, затем (по запросу) ревью кода с опорой на уточнённые docs.
3. **Критика содержания** — не только орфография и оформление: суть задач, подходов, слабые места, альтернативы с плюсами/минусами.
4. **Эволюция правил** — внутренние guidelines и этот документ улучшаются, если ревизия выявила системный пробел в правилах.
5. **Артефакты обязательны** — результат = зафиксированные находки + план гармонии, а не только устный разбор в чате.
6. **Объём по умолчанию** — тематический срез (строка AGENT-DOCS-INDEX / каталог / фича), не весь `docs/` за раз, если пользователь не попросил full-scan.
7. **Язык и стиль** — как в [docs-guidelines.md](./docs-guidelines.md); отклонения фиксировать как findings, не «молча править стиль всего дерева».

## 🧭 Фазы ревизии (порядок обязателен)

### Фаза 0 — Scope

| Шаг | Действие |
|-----|----------|
| 0.1 | Уточнить **границу**: тема / сервис / ADR / full `docs/` |
| 0.2 | Прочитать [AGENT-DOCS-INDEX](../00-meta/AGENT-DOCS-INDEX.md) по теме + этот файл (версия) |
| 0.3 | Зафиксировать **цель ревизии** одной фразой (например: «админ-карточка пользователей и related specs») |
| 0.4 | Перечислить **входной набор файлов** (обязательные + доп. по индексу) |

Без фазы 0 не переходить к правкам «на глаз».

### Фаза 1 — Качество самой документации

Проверить и зафиксировать findings по категориям:

| ID | Категория | Что искать |
|----|-----------|------------|
| D1 | **Коллизии** | Одно и то же утверждение в разных файлах с разным смыслом (имена сущностей, порты, SoT, ACL, статусы). Для auth-тем обязательно сверить peers: hard-lock · Logto suspend · rating ban · `X-Act-As` |
| D2 | **Несостыковки** | Битые/устаревшие ссылки, статус `stable` при TODO внутри, расхождение README ↔ ADR ↔ index |
| D3 | **Целостность** | Пропуски контракта (API без auth, entity без владельца schema, event без consumer), «висячие» TBD без владельца |
| D4 | **Логичность** | Неясный SoT, круговые зависимости описаний, противоречие продуктового слоя (`platform-for-users`) и tech-spec |
| D5 | **Стиль и оформление** | Emoji-заголовки, статусы, таблицы, шаблоны MICROSERVICE-SPEC, naming, язык — по [docs-guidelines](./docs-guidelines.md) |
| D6 | **Структура** | Документ не на своём месте, дубли вместо ссылок, ghost-планы вне DOCS-ROADMAP / AGENT-TODO |
| D7 | **Улучшения docs** | Что упростить, слить, вынести в ADR, сократить, добавить диаграмму/таблицу ответственности |

### Фаза 2 — Критический анализ сути

Не ограничиваться формой. Для описываемых задач и подходов:

1. Сформулировать **суть** подхода своими словами (1–3 предложения).
2. Подсветить **слабые / рискованные** места (безопасность, ops, DX, масштабирование, legal, fail-closed).
3. Предложить **1–3 варианта** улучшения/решения.
4. Для каждого варианта — **кратко плюсы и минусы** (и при возможности — рекомендуемый).
5. Отметить, где документация **маскирует неопределённость** словом «позже» / «TBD» без критерия выхода.

### Фаза 3 — Мета: правила и процесс

| Вопрос | Действие при «да» |
|--------|-------------------|
| Guidelines не покрывают найденный класс ошибок? | Предложить правку [docs-guidelines](./docs-guidelines.md) или naming |
| Этот процесс ревью дал сбой (пропуск фазы, размытый артефакт)? | Предложить правку **этого** файла (+ bump версии) |
| Index/roadmap вводят в заблуждение? | Обновить [AGENT-DOCS-INDEX](../00-meta/AGENT-DOCS-INDEX.md) / [DOCS-ROADMAP](../00-meta/DOCS-ROADMAP.md) |

Правила ревью **эволюционируют вместе с документацией** — не копить улучшения только в чате.

### Фаза 4 — Сверка docs ↔ код

Только после фиксации результатов фаз 1–3 (хотя бы в черновике ответа).

| ID | Категория | Что искать |
|----|-----------|------------|
| C1 | **Docs опережают код** | Spec есть, реализации нет / stub — статус в index должен это отражать |
| C2 | **Код опережает docs** | Поведение/endpoint/env есть в коде, в docs нет или устарело |
| C3 | **Рассинхрон** | Имена полей, пути API, порты, schema, роли Keto, feature flags — разные значения |
| C4 | **Ложные гарантии** | Docs пишут «✅ implemented», код/тесты не подтверждают |

Для каждого рассинхрона — **варианты решения** (обновить docs / изменить код / ADR / явно пометить docs-only) с плюсами/минусами.

### Фаза 5 — План гармонии

Свести в единый план (приоритет → действие → артефакт):

1. **P0** — ложные гарантии, security/ACL коллизии, опасный рассинхрон prod.
2. **P1** — целостность контрактов, SoT, index/status.
3. **P2** — стиль, структура, DX docs, дедуп.
4. **P3** — улучшения процессов/правил, «nice to have».

Формат пункта плана:

```text
[P?] ID · действие · файлы · критерий «готово»
```

План класть в ответ ревью; устойчивые пункты — в [DOCS-ROADMAP](../00-meta/DOCS-ROADMAP.md) и/или [AGENT-TODO.todo](../../AGENT-TODO.todo). Краткую запись прогона — в [DOCS-REVIEW-LOG.md](../00-meta/DOCS-REVIEW-LOG.md).

### Фаза 6 — Ревью кода (только по запросу)

После плана гармонии и **согласования**, что docs (или явные исключения) актуальны:

- Опираться на **обновлённую** документацию и общепринятые практики репозитория (Nest/Vue/pnpm, fail-closed, ADR).
- Качество кода должно **эволюционировать** вместе с проектом: findings → конкретные улучшения, без «переписать всё».
- Не смешивать фазы: code review ≠ молчаливое изменение docs mid-flight без фиксации C*/D*.

## 📤 Обязательный формат ответа агента

1. **Scope** — граница и список прочитанных файлов.  
2. **Findings (docs)** — таблица: ID · severity · файл · суть · предложение.  
3. **Критика подходов** — суть / риски / варианты ±.  
4. **Мета-правила** — что менять в guidelines / этом процессе (или «без изменений»).  
5. **Сверка код** — таблица C* · варианты решения ±.  
6. **План гармонии** — P0…P3.  
7. **Следующий шаг** — что делать сразу (патч docs / todo / ждать code review).

Severity: `blocker` · `major` · `minor` · `nit`.

Не начинать массовые правки docs/кода в той же сессии, пока пользователь не подтвердил план (исключение: явная просьба «сразу исправь»).
