# 📝 Правила ведения документации

> **Проект:** Tavrida Lot · **Версия guidelines:** 0.1

## 🎯 Принципы

1. **Язык** — русский, технические термины на английском (`BFF`, `JWT`, `TypeORM`).
2. **Имя проекта** — **Tavrida Lot** (не `max-alice`; это имя репозитория).
3. **Emoji в заголовках** — обязательны для навигации и единообразия.
4. **Статус** — каждый документ помечается: `draft`, `in progress`, `stable`.
5. **Минимум дублирования** — общее описывается в корневых разделах, детали — в сервисах.
6. **Два слоя продуктовой документации** — [platform-for-users.md](../01-goal/platform-for-users.md) (для людей) и технические specs (для реализации). При изменении фичи — обновлять оба.

## ✍️ Форматирование Markdown

- Inline-код: `` `parameterKey` `` — **без** экранирования (`\``).
- Блоки кода: языковой тег обязателен (`ts`, `json`, `http`, `bash`).
- Таблицы: выравнивание через `|`, заголовок с `---`.
- API-эндпоинты: блок `http` с методом и путём.
- Не дублировать блоки контента в одном файле.

## 🔄 Процесс обновления

1. Изменение **пользовательской** функциональности → [platform-for-users.md](../01-goal/platform-for-users.md) + [roles.md](../01-goal/roles.md) (если права) + технический spec.
2. Изменение **лимитов/настроек** → [PLATFORM-REGISTRY.md](../05-microservices/PLATFORM-REGISTRY.md) + README сервиса.
3. Изменение кода сервиса → обновить `docs/05-microservices/{service}/`.
4. Новый сервис → каталог + [docs/README.md](../README.md) + platform-for-users (если видно пользователю).
5. Архитектурное решение → `03-architecture/` + ADR.
6. PR с изменениями docs — ревью при изменении API, контракта или UX.
7. ADR / смена фазы → обновить [PROJECT-CONTEXT.md](../00-meta/PROJECT-CONTEXT.md).
